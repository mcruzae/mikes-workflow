01
Think · Diverge
Brainstorm in plan mode
No folders. No code. Just thinking.

Use plan mode (Shift+Tab in Claude Code, or the chat app) for divergent thinking. The goal is to explore the problem space — including which tech stack actually fits — not commit to anything yet.

Sample prompt
I want to build [type of project] that solves [problem] for [users]. Help me think through: - Who actually uses this and what pain it removes - Scope for v1 vs nice-to-haves - Deployment constraints (public, internal, on-prem, VPS, edge) - Candidate tech stacks worth considering for this specific use case, with brief pros and cons for each. Pick what suits the requirements. Do not default to anything. - Risks or unknowns worth surfacing now Don't write code or create folders yet. Just think out loud. Save the output as docs/brainstorm.md.
Why this matters: the stack question gets seeded in Step 1 but not decided until Step 2. This forces the LLM (and you) to consider real alternatives instead of reaching for the familiar one.
02
Think · Converge
Generate the PRD — commit stack & design language
Stack and design language are decided here.

Ask the LLM to translate the brainstorm into a structured PRD. Two sections matter most: Tech Stack (commits the framework) and Design Language (commits the aesthetic direction). Both decisions ripple through everything that follows.

Sample prompt
Read docs/brainstorm.md and produce a PRD in docs/PRD.md with these sections: - Problem (one paragraph) - Users (specific roles, not "the team") - Scope for v1 - Non-goals (what's explicitly out) - Success Criteria (observable, not aspirational) - Deployment Target - Tech Stack (with reasoning over alternatives) - Design Language (brutalist, Apple-like, editorial, Bloomberg/ Vision Pro premium dark, etc. — name the aesthetic and give 3 reference adjectives) - Tech Constraints (auth pattern, data residency, etc.) The Tech Stack and Design Language sections are commit points. Propose specifics with reasoning. Flag any decisions I should weigh in on. Keep the whole PRD under 2 pages.
Why design language belongs here: deciding the aesthetic early gives every design skill in the UI/UX section below a clear reference to work from, instead of starting from "make it look good" each time.
When to push back on the stack proposal
  • The reasoning is thin or hand-wavy
  • The choice optimizes for the LLM's familiarity rather than the project's needs
  • The stack is over-engineered for the scope
  • Deployment target and stack don't match
03
Scaffold · Baseline
git init + commit the docs
Clean baseline before any code exists.

Initialize the repo with only the planning docs. This gives a clean starting point you can always return to without losing the thinking work.

Manual commands
git init
git add docs/ && git commit -m "docs: brainstorm + PRD"
Or paste this prompt to do it all in one go
Set up version control for this project. 1. Initialize a git repo in the current directory (git init) 2. Add a .gitignore appropriate to the stack chosen in docs/PRD.md (Python, Node, etc.) plus the usual suspects: .env, .DS_Store, node_modules, __pycache__, etc. 3. Stage and commit the docs/ folder with the message "docs: brainstorm + PRD" 4. Ask me what to name the repo, then use the gh CLI to create a private GitHub repo under my account, add it as origin, and push the main branch Default to private. I can change it to public later with 'gh repo edit --visibility public' if needed.
Prerequisite: the gh CLI must be installed and authenticated once per machine (see Prerequisites box at the top). Run gh auth status to check.
Optional · Add a secret guard to prevent .env leaks

Once-per-machine setup. Catches accidental commits of .env files and API keys before they hit GitHub history (where they're nearly impossible to scrub).

Install a gitleaks pre-commit hook in this repo. 1. Check if gitleaks is installed (gitleaks version). If not, install it via brew (mac) or the appropriate package manager for my OS. 2. Create a .pre-commit-config.yaml at the repo root that runs gitleaks on every commit 3. Install pre-commit if not already installed (pip install pre-commit or brew install pre-commit) 4. Run 'pre-commit install' to wire it up 5. Run 'pre-commit run --all-files' once to verify it works Tell me if anything's missing or fails. After this, any commit that contains an .env file or detected secret will be blocked automatically.
04
Scaffold · Structure
Scaffold the folder structure from the PRD
The stack is settled. The structure follows.

The PRD has settled the stack — now the folder layout follows naturally from it. Different stacks imply different structures, and the LLM applies the right idioms for whichever was chosen.

Sample prompt
Read docs/PRD.md, specifically the Tech Stack section. Propose a folder structure idiomatic to that stack, then scaffold it. Include the stack's standard config files (package.json, requirements.txt, Cargo.toml, etc.) with sensible defaults and a starter README. Justify each top-level folder briefly in your response.
If on-prem is in play: ask the LLM to include a Dockerfile and docker-compose.yml from day one. Container-first means VPS-to-on-prem migration is a git pull and docker compose up regardless of the stack.
05
Scaffold · Memory
Invoke the external-memory skill
AGENTS.md becomes canonical. Cross-tool by design.

Run the claude-external-memory skill in init mode. It reads the manifest files created in Step 4 and the PRD, then produces a customized AGENTS.md, PLAN.md, PROGRESS.md, and VERIFY.md — plus pointer files for CLAUDE.md and GEMINI.md.

Run external-memory skill in init mode for this repo.
Why this beats running /init in each tool
  • Cross-tool — Claude Code, Codex, and Gemini CLI all read the same canonical source
  • Four files, four lifespans — norms, plan, log, and verification commands stay separate so they don't rot together
  • VERIFY.md captures the exact commands that prove the project works — the part /init skips entirely
  • Stack-aware — reads the manifest from Step 4 so the AGENTS.md actually reflects the real stack

Custom slash commands

Add a .claude/commands/ folder with repeatable commands tuned to this project. Each command is a single markdown file. The filename becomes the slash command (e.g. ship-check.md → /ship-check).

Don't write these by hand. Paste the prompt below into your AI assistant in VS Code and let it create the folder and file for you.

Copy-paste prompt to create a slash command
Create a custom slash command for this project. Command name: <name> What it should do: <describe the behavior in 2-3 sentences> Steps: 1. Create the folder .claude/commands/ if it doesn't exist 2. Create the file .claude/commands/<name>.md 3. Inside the file, write a clear prompt that tells Claude exactly what to do when /<name> is invoked 4. Add a one-line description at the top so it shows up in the slash menu After creating the file, tell me how to invoke it and remind me to restart the session so the command loads.
Current commands in this workflow: only /ship-check so far (see Step 9 for the full prompt). Add more as you notice yourself typing the same instruction repeatedly — that's the signal a command is worth creating.
VS Code workflow note: the .claude/commands/ folder lives in your repo and is picked up by the Claude, Codex, and Gemini VS Code extensions. Commit it to git so teammates inherit the same commands. For personal commands across all projects, use ~/.claude/commands/ instead.
07
Maintain · Session ritual
End each session with update mode
Keep the memory fresh while it's cheap.

Before closing a session, run the external-memory skill in update mode. It surveys recent commits, drafts a PROGRESS.md entry, ticks PLAN.md items, and adds new VERIFY.md checks if needed.

Run external-memory skill in update mode.
What gets written
  • A dated PROGRESS.md entry with one line per significant change
  • A Next: line pointing at the single most important next step
  • Updated checkboxes in PLAN.md
  • New smoke checks in VERIFY.md if endpoints or commands changed
Working conventions during the session

Branch per feature, even solo:

git switch -c feat/auth

Combined with git worktree, this lets you run parallel agent sessions on different features without context contamination.

Cold-start payoff: when you return to this project in 3 weeks, the next session starts with full context in under 30 seconds. No re-explaining.
08
Maintain · Drift audit
Periodically run check mode
Surface staleness before it becomes a problem.

Every few weeks, or after returning from a long break, run the skill in check mode for a deep staleness audit.

Run external-memory skill in check mode.
What it surfaces
  • PROGRESS.md entries older than 30 days
  • PLAN.md items pointing at files that no longer exist
  • VERIFY.md commands that reference missing binaries or scripts
  • Tech-stack drift between AGENTS.md and actual dependencies
  • Dangling Next: items that never made it to PLAN.md
09
Maintain · Ship
Run ship-check, then deploy
No deploy without a passing gate.

Two layers of pre-deploy verification: the design audit (impeccable, covered in the UI/UX section) audits the design itself, while the /ship-check slash command audits the implementation.

The ship-check ritual

A slash command at .claude/commands/ship-check.md that runs:

  • Responsive screenshots at 375px, 768px, and 1440px via Playwright MCP. Flags visible breakage.
  • Accessibility scan via axe-core on each key page. Reports serious/critical violations.
  • Lighthouse audit (for web projects) flagging any category below 80 — or 90 for public-facing sites.
  • VERIFY.md pre-commit section — every command in it must pass.
  • A single PASS/FAIL report. If FAIL, deploy stops.
Copy-paste prompt to create /ship-check
Create a custom slash command for this project called ship-check. Steps: 1. Create the folder .claude/commands/ if it doesn't exist 2. Create the file .claude/commands/ship-check.md 3. Inside the file, write a prompt that does the following when /ship-check is invoked: a) Take responsive screenshots at 375px, 768px, and 1440px using Playwright MCP, against pages listed in VERIFY.md b) Run axe-core accessibility scans on the same pages, flagging serious and critical violations c) Run Lighthouse (for web projects), failing if any category drops below 80 for internal tools or 90 for public-facing sites (read docs/PRD.md to determine which) d) Run every command in VERIFY.md's Pre-commit section e) Output a single Markdown report to /tmp/ship-check.md with a clear PASS/FAIL verdict f) If FAIL, stop the deploy and list the top 3 fixes needed 4. Add a one-line description at the top of the file so it shows up in the slash menu After creating the file, remind me to restart the session so /ship-check becomes available.
The universal deploy pieces — regardless of stack
  • Pre-deploy gate — ship-check must pass
  • Environment variables — never committed. Documented in docs/deploy.md.
  • Rollback command — defined and tested before you need it
  • Health check — a single command or URL that confirms the deploy worked

Reference stacks

Examples of what the workflow has produced — not defaults. Each project gets its own stack decision in the PRD.

Internal tool
Internal dashboards, automation, line-of-business utilities
LanguagePython
FrameworkFlask
DatabaseSQLite
AuthSession + Werkzeug
HostingHostinger VPS
Processsystemd + Gunicorn
Reverse proxyNginx + Let's Encrypt
Path convention/opt/<project>
Why this stack
  • Simpler mental model. One language, one process, one file for the database. No build step, no hydration, no client/server split.
  • SQLite is enough. Internal tools rarely exceed 100 concurrent users. SQLite handles that easily, no separate database service to install, secure, or back up. Backup is a file copy.
  • Faster to v1. Empty repo to deployed in a couple of hours. No auth setup, no ORM config, no deployment dashboard.
  • Cheap to run. Runs comfortably on a low-tier VPS. No managed-database monthly bill.
  • Easy to debug. Python tracebacks point at the actual problem. No "is it server or client?" confusion.
  • Mature for AI-assisted dev. 15+ years of Flask patterns. LLMs write idiomatic Flask reliably.
Public web app
Marketing sites, content-heavy apps, SEO-driven projects
LanguageTypeScript
FrameworkNext.js
DatabasePostgres
ORMPrisma or Drizzle
HostingVercel or VPS
ProcessNode + PM2 (on VPS)
Reverse proxyVercel edge / Nginx
Type safetyEnd-to-end
Why this stack
  • SEO is first-class. Server-side rendering, metadata APIs, sitemap generation — public sites need this and Next.js delivers it out of the box.
  • Built for public traffic. Edge rendering, ISR, image optimization, automatic code splitting. Performs well under real load without manual tuning.
  • Postgres handles real scale. Concurrent writes, full-text search, JSON queries, replication. SQLite serializes writes and falls over at scale.
  • Type safety end-to-end. TypeScript from the database (via Prisma/Drizzle) through the UI catches whole classes of bugs at compile time. Pays off as the codebase grows.
  • Rich UI ceiling. React beats server-rendered templates when the interface needs to be highly interactive — real-time updates, complex forms, drag-and-drop.
  • CDN-friendly. Designed for global edge deployment. Critical when users span multiple regions.
When to switch sides
Internal tool → upgrade path before rewriting
  • SQLite → Postgres: first stop. Swap the database, keep Flask. Handles 1000+ concurrent users.
  • Reach for Next.js only if: UI needs to be highly interactive (real-time, complex state)
  • Or: the team already knows TypeScript better than Python
  • Or: compliance demands audit-grade DB with replication and Postgres alone isn't enough
  • A full rewrite is rarely the right answer for "more users" — upgrade the database first.
Public app → reach for Flask/FastAPI if
  • It's an API-only service with no UI
  • Backend does heavy data processing (pandas, ML, scientific)
  • Sitting behind a Next.js frontend as a microservice
  • The team has deep Python expertise and no TypeScript

Other stacks (Go, Rust, FastAPI, SvelteKit, static sites) follow the same workflow — only the deployment ritual changes.

What good output looks like

Short examples of the artifacts the workflow produces. Useful as a reference when something the LLM generates doesn't quite feel right.

docs/PRD.md
Tech Stack section — what good reasoning looks like
## Tech Stack

**Recommendation:** Python + Flask + SQLite, deployed to Hostinger VPS.

**Reasoning:**

- Expected users: 8 to 12 internal staff across 3 country offices.
  Concurrent writes will rarely exceed 5. SQLite handles this easily.
- No public-facing requirement. SEO and edge rendering are not needed,
  which removes the main reasons to choose Next.js.
- The team is more comfortable with Python than TypeScript. Choosing
  Flask shortens the path to v1 by an estimated 2-3 weeks.
- Auth requirements are simple (internal SSO via existing IdP).
  Session-based auth with Werkzeug covers this.

**Alternatives considered:**

- Next.js + Postgres: rejected for v1 due to over-engineering for the
  user count. Worth revisiting if scope expands to external partners.
- Django: rejected because the admin UI it provides isn't needed and
  the framework overhead doesn't pay off at this scale.

**Decisions for Mike to weigh in on:**

- Should we use Postgres instead of SQLite from day one to make a
  future migration easier? My recommendation: no, SQLite is fine.
- Should we plan for Docker now? My recommendation: yes if on-prem
  deployment is a realistic future, no if Hostinger is final.

Good signals: specific numbers, explicit alternatives, named trade-offs, and questions back to the user. A bad PRD section says only "Python + Flask, because it's simple."

AGENTS.md
Project conventions section — concrete, not vague
## Conventions

**Writing style (applies to all generated text — commits, docs, emails):**
- Plain English. No em dashes or en dashes under any circumstances.
- Dates expressed with "to" not hyphens (e.g. "2024 to 2026").
- No language that sounds AI-generated.

**Code style:**
- Python: ruff for linting, black for formatting, line length 100.
- Routes in src/routes/, one file per feature area.
- Database access goes through src/db.py — no direct SQL elsewhere.
- Environment variables read once at startup via src/config.py.

**Commit messages:**
- Format: type(scope): description
- Types: feat, fix, docs, refactor, chore, test
- Example: feat(auth): add session timeout after 30 minutes idle

**Things to never do:**
- Never commit .env files or any file matching .env.*
- Never disable the pre-commit secret guard
- Never use SQLite WAL mode without checking with Mike first

Good signals: specific rules with examples, not principles. A bad AGENTS.md says "write clean code" or "follow best practices." This one tells the agent exactly what to do.

PROGRESS.md
One session's entry — written at session close, not weeks later
## 2026-05-22 · Session 14

**Done:**
- Added session timeout (30 min idle) in src/auth.py
- Migrated user table to include last_active_at timestamp
- Wrote 4 tests for session expiry edge cases (all passing)
- Fixed bug where /logout didn't clear the session cookie on Safari

**In progress:**
- Email notification when a session is forcibly ended

**Next:** Wire up the email send via the existing SMTP relay
configured for the bulk mailer. See src/mailer.py for the pattern.

**Notes for future me:**

Good signals: specific files touched, the single "Next:" pointer, and "Notes for future me" capturing tribal knowledge. A bad PROGRESS.md entry says "worked on auth stuff" with no specifics.

What's coming next — short, current, actionable
# PLAN

## Now (this week)
- [x] Session timeout implementation
- [x] Safari cookie bug fix
- [ ] Email notification on session end

## Next (after current)

Operating principles

Defaults that make the workflow worth the discipline.

01
Stack is an output, not an input
The PRD decides the stack based on the project's real requirements. The workflow doesn't favor any framework.
02
Design language is decided in the PRD
Aesthetic direction is locked alongside the tech stack. Every design skill points at it instead of starting from scratch.
03
Docs before code
Brainstorm, PRD, and folder structure all exist as documents before a single feature is written.
04
Canonical, not duplicated
AGENTS.md is the source of truth. CLAUDE.md and GEMINI.md are three-line pointers. Never duplicate.
05
Update at the boundary
PROGRESS.md gets updated at the end of every session — not weeks later when memory has faded.
06
Responsive and accessible by default
Ship-check runs before every deploy, covering responsive layout, accessibility, and Lighthouse scores. Catching issues at the gate is cheaper than catching them in production.
07
Verify, don't assume
VERIFY.md exists so that "it works" is observable. Every project ships with exact commands that prove it.
08
Capture the ritual as a command
If you find yourself typing the same instruction often, save it as a slash command. One markdown file in .claude/commands/, available to every AI assistant in VS Code.
09
Plain English, always
No em dashes, no en dashes. Concise writing for senior stakeholders. Language that doesn't sound AI-generated.
10
Containerize when in doubt
If a project might move between VPS, on-prem, or edge, Docker is the abstraction that makes one repo work everywhere.
Glossary

Short definitions for the terms used throughout this workflow. Scan it if something earlier on the page felt opaque.

When to read this: only if a term above didn't make sense. This is a reference, not required reading. Italicized terms are cross-references to other entries here.
AI tooling
MCP (Model Context Protocol)
A standard for connecting AI assistants to external tools and data. An MCP server (like Hostinger, GitHub, or Playwright) gives Claude or Codex the ability to interact with that service directly from a chat.
Skill
A reusable instruction set that teaches an AI assistant how to do something specific. Lives in a folder (e.g. .claude/skills/<name>/SKILL.md) and gets triggered automatically when the AI detects a matching context.
Slash command
A custom shortcut you invoke by typing /name in your AI chat. The command's logic lives in a single markdown file at .claude/commands/<name>.md. Simpler than a skill — one file, no folder.
AGENTS.md
The canonical project context file that AI assistants read on every session. Contains stack info, conventions, and rules. Both Claude and Codex read it by default; CLAUDE.md and GEMINI.md can be three-line pointer files that redirect to it.
Plan mode
A Claude Code feature (Shift+Tab to toggle) where the AI proposes a plan before executing anything. Used in Step 1 for brainstorming without the AI jumping straight to code.
Workflow files
PRD (Product Requirements Document)
A short doc that defines what a project is, who it's for, what's in scope, and what stack will build it. Generated in Step 2. Becomes the single source of truth that downstream steps read from.
VERIFY.md
The exact commands that prove a project works — start command, smoke tests, pre-commit checks, pre-deploy checks. Created by the external-memory skill and run by /ship-check.
PROGRESS.md
A dated log of what got done in each session. Updated at session close by the external-memory skill in update mode. Gives cold-start sessions full context in seconds.
PLAN.md
What's coming next on the project, organized as Now / Next / Later / Blocked. Pruned regularly so it stays under a dozen items.
Stack & framework terms
ORM (Object-Relational Mapper)
A library that lets you query a database using your programming language instead of raw SQL. Prisma and Drizzle are the two common choices for TypeScript projects.
ISR (Incremental Static Regeneration)
A Next.js feature that serves pre-rendered pages but refreshes them in the background on a schedule. Gives static-site speed with fresh content.
Edge rendering
Serving content from CDN locations geographically close to the user instead of one central server. Much faster for global audiences. Built into Vercel and Cloudflare.
WAL mode (Write-Ahead Logging)
A SQLite configuration that allows concurrent reads while a write is happening. Makes SQLite usable for slightly higher-traffic apps, but has its own gotchas — don't enable without testing.
Container / Docker
A way to package a project so it runs identically anywhere — local laptop, VPS, on-prem server. The Dockerfile defines the package; docker-compose.yml runs it. Key for projects that might move between hosting environments.
Deployment & tooling
gh CLI
GitHub's command-line tool. Lets you create repos, manage PRs, and authenticate from the terminal. Step 3 uses it to create a private repo and push automatically.
systemd
The standard Linux service manager. Used on VPS deployments to keep a Flask/Gunicorn process running, auto-restart on crash, and start on boot.
PM2
A Node.js process manager. Same job as systemd but for Node apps — keeps the Next.js server running, restarts on crash, manages logs. Used when deploying Next.js to a VPS instead of Vercel.
Reverse proxy
A server that sits in front of your app and routes incoming requests. Nginx is the standard choice — handles HTTPS, caching, and routing multiple apps on one VPS. Lets you run several projects on different subdomains from the same machine.
Playwright
A browser automation library. The Playwright MCP lets AI assistants take screenshots, run interactions, and test pages. Used by /ship-check for responsive testing.
axe-core / Lighthouse
Two automated quality tools. axe-core scans pages for accessibility violations (contrast, missing labels, etc). Lighthouse scores performance, SEO, best practices, and accessibility. Both run inside /ship-check.
gitleaks
A pre-commit hook that scans for secrets (API keys, tokens, passwords) before they get committed. Once installed, it blocks any commit containing a detected secret. Strongly recommended for any repo that might go to a public host.

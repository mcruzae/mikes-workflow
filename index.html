01
Think · Diverge
Brainstorm in plan mode
No folders. No code. Just thinking.

Use Claude's plan mode (Shift+Tab in Claude Code, or the app's planning interface) for divergent thinking. The goal here is to explore the problem space, not commit to it.

What to capture
  • The actual problem this internal tool solves
  • Who uses it (D-Link MEA team, country offices, specific roles)
  • Adjacent tools or workflows it replaces or connects to
  • Constraints worth surfacing early (compliance, regional latency, on-prem vs VPS)
Why no folders yet: committing to structure during brainstorming makes implicit architecture decisions. Flask vs FastAPI dictates one layout, static site another. Stay loose.
Output

A single file: docs/brainstorm.md — or just a saved chat thread.

02
Think · Converge
Generate the PRD / blueprint
Turn the brainstorm into a real spec.

Ask Claude to translate the brainstorm into a structured Product Requirements Document. This is where ambiguity gets killed.

Sections worth including
  • Problem — the one-paragraph reason this exists
  • Users — specific roles, not "the team"
  • Scope — what's in v1
  • Non-goals — what's explicitly out (this saves the most time later)
  • Success criteria — observable, not aspirational
  • Deployment target — Hostinger VPS, on-prem, or undecided
  • Tech constraints — Flask default, SQLite default, internal-only auth pattern
Read docs/brainstorm.md and produce a PRD in docs/PRD.md following the sections: Problem, Users, Scope, Non-goals, Success Criteria, Deployment, Tech Constraints. Keep it under 2 pages.
03
Scaffold · Baseline
git init + commit the docs
Clean baseline before any code exists.

Initialize the repo with only the planning docs. This gives a clean starting point you can always return to without losing the thinking work.

git init
git add docs/ && git commit -m "docs: brainstorm + PRD"
Add a pre-commit secret guard now, before any code exists. Tools like gitleaks or a simple regex pre-commit hook prevent .env leaks that are nearly impossible to scrub from history later.
Repo origin

Push to a private repo before writing code. Future-you wants the rollback option.

04
Scaffold · Structure
Propose & scaffold the folder structure
Ask Claude Code to derive it from the PRD.

Now structure earns its place — the PRD has settled the tech stack and components, so the folder layout follows naturally.

Read docs/PRD.md and propose a folder structure. Justify each top-level folder. Then scaffold it.
Typical layout for internal Flask tools
/src · /tests · /docs · /scripts · /static · /templates · .env.example · requirements.txt
Decide Docker now if on-prem is a possibility. A docker-compose.yml from day one means VPS-to-on-prem migration is a git pull and docker compose up — not a rebuild.
05
Scaffold · Memory
Invoke the external-memory skill
AGENTS.md becomes canonical. Cross-tool by design.

Run the claude-external-memory skill in init mode. It reads the README, package files, and git log to produce a customized AGENTS.md, PLAN.md, PROGRESS.md, and VERIFY.md — plus pointer files for CLAUDE.md and GEMINI.md.

Why this beats /init
  • Cross-tool — Claude Code, Codex, and Gemini CLI all read the same canonical source
  • Four files, four lifespans — norms, plan, log, and verification commands stay separate so they don't rot together
  • VERIFY.md captures the exact commands that prove the project works — the part /init skips entirely
Run external-memory skill init on this repo
Why invoke after scaffolding: the skill reads package.json, requirements.txt, and the README to seed the tech-stack table. Invoking before those exist produces stubs full of _TODO_ markers.
06
Scaffold · Tooling
Wire up MCP servers & slash commands
Make agents project-aware from day one.
MCP servers to consider
  • Hostinger MCP — VPS state, disk, ports, services. Stops Claude from suggesting port 5000 when the mailer already owns it.
  • Playwright MCP — UI smoke tests for internal dashboards
  • Postgres / SQLite MCP — if the project has a database, let agents inspect schema directly
  • GitHub MCP — for PR review and issue triage workflows
Custom slash commands

Add a .claude/commands/ folder with repeatable commands tuned to this project:

/add-feature → scaffolded route + template + test stub
/review-pr → walks recent diff against AGENTS.md conventions
/deploy → runs verify checks then ships to VPS
07
Build · Iterate
Vibe-code with Claude Code + Codex in parallel
Each tool has a different strength. Use both.
Division of labor
  • Claude Code — architecture, refactoring, code review, debugging the hairy stuff
  • Codex — focused implementation tasks, test writing, boilerplate
  • Cross-check pattern — Codex implements, Claude Code reviews against AGENTS.md conventions
Branch per feature, even solo
git switch -c feat/auth

Combined with git worktree, this lets you run parallel agent sessions on different features without context contamination.

Plain English, no em dashes, concise. Bake this into AGENTS.md so every drafting task — emails, docs, commit messages — defaults to your house style.
08
Maintain · Session ritual
End each session with update mode
Keep the memory fresh while it's cheap.

Before closing a session, run the external-memory skill in update mode. It surveys recent commits, drafts a PROGRESS.md entry, ticks PLAN.md items, and adds new VERIFY.md checks if needed.

Run external-memory skill update
What gets written
  • A dated PROGRESS.md entry with one line per significant change
  • A Next: line pointing at the single most important next step
  • Updated checkboxes in PLAN.md
  • New smoke checks in VERIFY.md if endpoints or commands changed
Cold-start payoff: when you return to this project in 3 weeks, the next session starts with full context in under 30 seconds. No re-explaining.
09
Maintain · Drift audit
Periodically run check mode
Surface staleness before it becomes a problem.

Every few weeks, or after returning from a long break, run the skill in check mode for a deep staleness audit.

Run external-memory skill check
What it surfaces
  • PROGRESS.md entries older than 30 days
  • PLAN.md items pointing at files that no longer exist
  • VERIFY.md commands that reference missing binaries or scripts
  • Tech-stack drift between AGENTS.md and actual dependencies
  • Dangling Next: items that never made it to PLAN.md
Doesn't auto-fix. Surfaces findings as a numbered list and asks before changing anything. The judgment stays with you.
10
Maintain · Deploy
Ship to VPS with the deploy ritual
Same pattern across every project.

Standardize the deployment pattern in a docs/deploy.md and reference it from AGENTS.md. The proven D-Link MEA pattern:

VPS path convention
/opt/<project-name>
Service layout
  • systemd service for the Flask/Gunicorn process
  • Nginx reverse proxy with Let's Encrypt
  • Environment vars via systemd EnvironmentFile, never committed
  • SQLite under /opt/<project>/data/ with nightly backup cron
Pre-deploy gate

Run every command in VERIFY.md's Pre-commit section. If anything fails, deploy stops.

If on-prem is in play: Docker Compose is the abstraction that makes both targets work from the same repo. Hostinger pulls the same image on-prem would.
Operating principles

Defaults that make the workflow worth the discipline.

01
Docs before code
Brainstorm, PRD, and folder structure all exist as documents before a single route is written.
02
Canonical, not duplicated
AGENTS.md is the source of truth. CLAUDE.md and GEMINI.md are three-line pointers. Never duplicate.
03
Update at the boundary
PROGRESS.md gets updated at the end of every session — not weeks later when memory has faded.
04
Verify, don't assume
VERIFY.md exists so that "it works" is observable. Every project ships with exact commands that prove it.
05
Skills earn their place
A skill exists when a ritual repeats with non-obvious logic. Otherwise it's premature optimization.
06
Plain English, always
No em dashes, no en dashes. Concise writing for senior stakeholders. Language that doesn't sound AI-generated.
